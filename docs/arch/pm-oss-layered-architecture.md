# PM 系 OSS のレイヤー分割

## 目的

自前実装と OSS 借用の境界を、構造レベルで先に固定する。

この文書は、今後の設計で「どこまで自前で握るか」「どこを参照・借用候補にするか」の判断基準に使う。

## 基本方針

- プロダクトの差別化要因は入口から整理までの一気通貫に置く
- そのため、入口と AI 分別の中核は自前実装に寄せる
- 一方で、ガント、スケジュール、横断ビューのような高コスト可視化は OSS 借用候補とする
- OSS はアプリ全体採用ではなく、レイヤー単位で扱う

## レイヤー分割

### 1. 入口レイヤー

責務:
- アイデア入力
- 外部入口の回収
- AI 分別
- タスク化前のレビュー
- Codex プロジェクト開始支援

扱い:
- 自前で握る

理由:
- ここが今回プロダクトの中核価値だから
- Slack / knowledge-vault / AI 分別の接続は既存 OSS にそのまま載せにくいから

### 2. 業務オブジェクトレイヤー

責務:
- アイデアカード
- タスク
- 予定
- 人
- ロール
- 検討事項
- 気になっている事
- 外部入口イベント

扱い:
- 自前で握る

理由:
- 複数 OSS をまたいで一貫したモデルを保つ必要があるから
- Codex 起点のプロジェクト開始支援も同じデータモデルに寄せたいから

### 3. 計画・可視化レイヤー

責務:
- タスクリスト表示
- TODO 表示
- ガント表示
- スケジュール表示
- 人員計画表示
- 横断ダッシュボード表示

扱い:
- OSS 借用候補

主参照:
- `OpenProject`
- `Leantime`
- `Plane`

### 4. 権限・公開範囲レイヤー

責務:
- ロール別の可視範囲制御
- 外部協力者向け制限
- 上位管理者向け操作制御

扱い:
- 自前で握る

理由:
- 4 窓口全体にまたがる制御なので、特定 OSS に依存しすぎると破綻しやすいから

## OSS ごとの担当レイヤー

### `Plane`

担当させるレイヤー:
- 入口レイヤーの情報設計参照
- ダッシュボードの情報整理感の参照

担当させないレイヤー:
- ガント主構造
- 人員計画主構造

### `OpenProject`

担当させるレイヤー:
- 計画・可視化レイヤーの構造参照
- タスクとガントの一体オブジェクト設計参照
- 横断計画ビューの参照

担当させないレイヤー:
- 入口 UX の主軸
- 自前の軽量入力体験

注意:
- `Team planner` は Enterprise add-on なので、実装流用ではなく構造参照として扱う

### `Leantime`

担当させるレイヤー:
- TODO / task list の体験参照
- ideas から todos への昇格導線の参照
- 軽量な進行体験の参照

担当させないレイヤー:
- 横断管理の主構造
- 権限や大規模運用の主軸

## 4 窓口との対応

### 横断ダッシュボード

- 主構造: `OpenProject`
- 補助参照: `Plane`

### 上位管理者用の管理画面

- 主構造: 自前
- 補助参照: `OpenProject` の権限・横断管理の考え方

### 書き入れ口 / 作成口

- 主構造: 自前
- 補助参照: `Plane` と `Leantime`

### 実績・履歴参照

- 主構造: 自前
- 補助参照: `OpenProject` の履歴的な見せ方を参考にする余地あり

## 実装に持ち込む時の判断基準

1. そのレイヤーが差別化要因なら自前に寄せる
2. そのレイヤーが可視化コストの高い共通課題なら OSS 借用を優先する
3. OSS 借用時は `Web で閲覧可能` と `フォークしやすさ` を必須条件に近く扱う
4. ライセンス差や版差がある機能は、構造参照と実装流用を分けて判断する

## 現時点の仮結論

- 自前で握る中心は `入口 + AI 分別 + 業務オブジェクト + 権限`
- `OpenProject` は `ガント + 横断計画構造` の主参照
- `Leantime` は `TODO + ideas 昇格体験` の主参照
- `Plane` は `入口の情報設計` の主参照

この分割で進めると、要件が増えても「どの OSS をどこに効かせるか」を整理しやすい。
