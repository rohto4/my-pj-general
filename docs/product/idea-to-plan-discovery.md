# やりたいこと抽出フェイズ 初版

## 目的

このプロダクトで今後一元管理したいことを整理し、MVP 候補と後回し候補を分ける。

## 現時点で見えている中心価値

- 雑なやりたいことをアイデアカードとして受け止める
- Slack や knowledge-vault からも情報を取り込む
- 具体化したいタイミングでタスク化できる
- AI 補完により自動タスク化できる
- タスクを Google Calendar 登録までつなげる
- 複数人・複数タスク前提で進行状況を可視化する
- 人ごとのキャパ超過を把握できる
- 外部協力者にも限定権限で使わせられる

## やりたいこと一覧 初版

- アイデアカードを雑に入れる
- アイデアカードを後から具体化する
- アイデアからタスクを手動または AI で生成する
- タスクにジャンルを持たせる
- タスクを担当者・期限・優先度・見積・ステータスで管理する
- タスクを Google Calendar 予定登録につなげる
- 複数人で動くタスク群を一覧で追う
- 複数タスクの並行進行を把握する
- ガントチャートで進行を見える化する
- 人員キャパ超過を検知する
- 外部協力者がチケット消化や予定確認を行える
- 将来は社内メンバーにも同じ基盤を広げる
- Codex で新規プロジェクトを始めるときの初期セットアップを半自動化する
- 初期プロンプト、フォルダ設計、テンプレート流用元の選定を毎回手書きしなくて済むようにする
- Slack や knowledge-vault の投稿を定期回収し、AI で自動分別して登録する

## 画面・窓口の大分類 初版

### 1. 横断ダッシュボード

- 全体のタスクを横断表示する
- 全体のスケジュールを横断表示する
- 人員キャパを横断表示する
- ロールに応じて見える範囲を制御する
- 主目的は閲覧と状況把握であり、強い管理操作はここに置かない
- 情報設計や見せ方は、既存の PJ 管理 OSS にできるだけ従う
- 独自UIを優先するのではなく、既存 OSS の洗練された構造を参考にする

### 2. 上位管理者用の管理画面

- ダッシュボードからは操作できない管理系操作を扱う
- あまり機能過多な印象にはしない
- PJ リーダーが通常操作しない、またはできない処理を扱う
- 例:
- 権限やロールの調整
- 組織・人員設定
- 全体ルール変更
- 例外的な調整操作

### 3. 書き入れ口 / 作成口

- アイデアの入力
- タスクの入力
- アイデアからタスクへの昇格
- AI 補完
- プロジェクト作成
- Codex 用プロジェクト開始支援
- Slack / knowledge-vault から回収した項目のレビューと確定
- UI は自前実装を前提にする
- UI 基盤候補として `shadcn/ui` を使う

### 4. 実績・履歴参照

- 過去の実績を見る
- 過去のタスク、進行、完了状況を見る
- 過去のスケジュール実績を見る
- 将来的には判断記録やナレッジ接続の入口にもなりうる
- まずは実績データとステータス履歴を十分に持つ
- 見せ方は後から拡張できる前提で、初期はデータ設計を優先する

## 現時点の仮説

- 起点は「タスク管理ツール」ではなく「アイデアを雑に入れられる入口」である
- 入口は手入力だけでなく、Slack と knowledge-vault の定期回収も含む
- 価値の中核は、入力からタスク化と予定化までの断絶をなくすことにある
- 初期フェーズでは、ガントやキャパ管理は欲しいが、主価値を成立させるための必須要件ではない可能性が高い
- ロール管理は初期から必要だが、権限体系そのものは最小限から始めるのが妥当
- このサイトは、仕事管理だけでなく Codex 用プロジェクト起動支援も担う
- UI は単一画面ではなく、閲覧・管理・入力・実績参照の 4 窓口に分けるのが自然
- ダッシュボードの情報設計は既存 OSS を参考にした方がよい
- 実績画面は、初期は可視化よりも履歴データ保持の設計が重要
- knowledge-vault は AI とのやり取りの中で自然更新される前提なので、明示入力UIよりも回収と再分類の導線が重要
- 入口UIは OSS プロダクト流用ではなく、自前実装の方がよい
- 入口UIの見た目と部品基盤には `shadcn/ui` が有力

## 2026-06-23 時点の OSS 仮説

- 情報入口やモダンな PM 体験の参照先としては `Plane` がかなり強い
- `ガント + TODO / タスク` の一体 OSS としては `OpenProject` と `Leantime` が強い
- 横断管理、権限、人員計画まで含めた主参照先としては `OpenProject` が優勢
- `ideas -> todos -> gantt` の流れや TODO デザイン参考としては `Leantime` が有力
- 詳細は `docs/candi-ref/oss-candidates-intake-pm.md` と `docs/candi-ref/openproject-vs-leantime-integrated-comparison.md` を参照
- 現時点の本線仮説は `入口自前 + Plane + OpenProject / Leantime からの選択的借用`

## 2026-06-23 時点の設計方針仮説

- 情報の入口は自前で作る
- アイデアカード、AI 補完、タスク化導線、ロール設計も自前の中核機能として扱う
- ただし、ガントチャートやスケジュール表示のように視覚表現の完成度が重要な領域は、OSS の力を借りる前提で考える
- ガントチャートは `OpenProject` を主参照、`Leantime` を補助参照にして検討する
- タスクリスト / TODO リストもデザイン品質が重要なので、`Leantime` を中心に OSS の部分取り込みを濃いめに視野に入れる

## この方針の意味

- プロダクトの差別化要因は「入口から整理までの一気通貫」に置く
- OSS はプロダクト全体の土台としてではなく、表示・可視化・一部操作体験の加速材料として扱う
- そのため、候補 OSS は「丸ごと採用するか」ではなく「どのレイヤーを借りるか」で評価する

## レイヤー分割の初版

### 自前で握る層

- アイデアカード入力
- AI による要約、補完、タスク案生成
- アイデアからタスクへの昇格
- Slack / knowledge-vault からの回収と AI 分別
- 入口UIそのもの
- ジャンル、担当、期限、優先度、依存、公開範囲などの業務データモデル
- ロール管理と権限設計
- Google Calendar 連携方針
- Codex プロジェクト起動用のプロンプト生成
- 初期フォルダ構成とテンプレート選定ロジック

### OSS 借用候補の層

- タスクリスト描画
- TODO リスト描画
- ガントチャート描画
- スケジュールビュー描画
- タイムライン上のドラッグ操作
- 負荷可視化のビュー部品
- ただし前提条件として、Web で閲覧可能であること
- OSS としてフォークしやすく、デザインや挙動を弄りやすいこと
- `OpenProject` はガントと人員計画の構造参照先として扱う
- `Leantime` は ideas / TODO / task list 体験の参照先として扱う

## 次に詰めるべき論点

- OSS を「アプリ丸ごと」ではなく「ライブラリや一部画面実装の参考」として借りるか

- 表示だけ借りるのか、編集操作まで借りるのか
- Codex プロジェクト起動支援を、タスク管理と同一画面で扱うか、別モードで扱うか
- 横断ダッシュボードでどこまで操作を許可するか
- 実績画面で何を比較対象として見たいか
- 横断ダッシュボードの参照元として、どの OSS の思想を採るか
- Slack と knowledge-vault の回収対象をどの粒度で拾うか
- AI 自動分別の確定前に人がレビューするか、即登録するか
- ガント候補は `Web 閲覧可能` と `フォークしやすさ` を必須条件にするか
- `OpenProject` のどのレイヤーまで借りるか
- `Leantime` のどのレイヤーまで借りるか
