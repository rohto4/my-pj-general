# コンテキスト圧迫・セッション切替ガイド

## 目的

長い会話を「文字数」だけで判断せず、実際にモデルへ渡る入力 token、圧縮回数、再送される画像・添付・巨大 tool 出力を分けて扱う。圧縮後でも安全に実装を続けるか、docs と Git の断面を閉じて新しいセッションへ移るかを一貫して決める。

この文書は採用済みの運用手順の正本である。調査の実測根拠は `docs/imp/context-pressure-investigation-2026-07-13.md`、構造統計の再取得は `.agents/skills/audit-context-pressure/` を使う。

## 判断に使う値

| 値 | 用途 | 判断に使わない誤用 |
| --- | --- | --- |
| `last_token_usage.input_tokens / model_context_window` | 次の応答に載る実入力の占有率 | 累積 token 使用量を現在の占有率と扱うこと |
| 圧縮回数 | 置換履歴が何度積み重なったか | 圧縮1回だけで必ず失敗と断定すること |
| `input_image` と添付の数・構造サイズ | 高コスト payload が圧縮後も残る兆候 | JSON byte をモデル token と同値扱いすること |
| 大きい tool 出力・全画面ログ | 画像以外の圧迫要因 | 種別別 raw byte を加算して総量と扱うこと |
| 初期読込の token 実測 | 起動時の基礎負荷 | session 全体の圧迫原因と短絡すること |

## 状態と行動

実行時に `model_context_window` が取得できる場合は、その値を分母にする。取得できない場合は、圧縮回数と payload の種類を優先し、推測の百分率を記録しない。

| 状態 | 目安 | 次の行動 |
| --- | --- | --- |
| 緑: 通常 | 実入力が60%未満、圧縮0回、画像・長大出力が少ない | 現セッションで続行する。タスク・正本は通常どおり更新する。 |
| 黄: 断面準備 | 実入力60〜74%、圧縮1回、または画像／全ログ出力を増やした | 現在の小単位をテスト・docs・タスクへ同期する。次に画像監査や大規模探索へ入る前に新セッションを選ぶ。 |
| 橙: 切替推奨 | 実入力75〜84%、圧縮1回後に高コスト局面が続く、または画像・添付が圧縮置換履歴に残る | 新規実装や全画面レビューを始めない。Git 断面、`docs/imp/*`、必要なら diary を更新して新セッションへ移る。 |
| 赤: 切替必須 | 実入力85%以上、圧縮2回以上、または圧縮後も多数の画像・巨大出力が残る | 同一セッションで大きな作業を続けない。安全な断面を閉じ、新セッションで必須初期化から再開する。 |

実入力の割合が一時的に下がっても、圧縮置換履歴に画像 payload が残っている場合は緑へ戻したとみなさない。新しいセッションだけがその履歴を外す。

## 作業パターン別の扱い

### 実装先行で設計書が薄い場合

1. 対象機能だけを特定し、変更した source、テスト、画面・API境界を列挙する。
2. `docs/imp/imp-tasks.md` に不足設計を記録し、`docs/product` / `docs/spec` / `docs/data` / `docs/arch` の責務へ最小単位で補う。
3. source 全体や過去 diary を読み直して圧迫を補おうとしない。設計書化は機能単位で閉じる。

### 設計書を先に整えている場合

1. `docs/guide/implementation-context-reading-guide.md` で役割を選ぶ。
2. 対象見出し・対象受入ID・不足証拠種別だけを読む。リンクは自動・再帰読込キューにしない。
3. 1ファイルが長い場合は対象節を先に読み、節だけで断定できない場合に限り範囲を広げる。

### 実装と設計書が混在する場合

現在の機能に必要な「体験、状態/API、データ、外部境界、運用、回帰」を一行ずつ対応付ける。未読の別機能、全ソース、HTML全体を読んだ量で埋めない。

### 画像・ログ・デバッグが中心の場合

- 一回の判断では代表スクリーンショットを絞る。比較が終わった画像生成案や全画面JPEGを、次の実装局面へ持ち越さない。
- tool は対象ID、件数、失敗行、必要な前後だけを返す。全文ログや大きなJSONはファイルへ保存し、必要箇所だけを読む。
- 画像生成、複数スクリーンショット、巨大 tool 出力の後は、その結果を docs へ確定してからセッションを切る。

## 新セッションへ渡す最小断面

1. `docs/imp/imp-tasks.md`: 完了、未完、次の一手、実データ変更の有無。
2. `docs/imp/imp-comp.md`: 完了した最小単位と検証結果。
3. 必要な場合だけ `docs/diary/*`: 次回に読む正本、コマンドではなく安全境界、未解決理由。
4. Git: テスト済みで目的が一つの変更だけを commit する。未検証の別目的変更は混ぜない。
5. 新セッション: `AGENTS.md` の必須初期化と、作業役割に対応する最小読込セットから始める。

## 調査の安全境界

セッションログを分析する場合も、本文、画像内容、Cookie、token、secret、envを表示・転記しない。構造種別、件数、サイズ、token メタデータだけを使い、raw byte は token の代替値ではないと明示する。
