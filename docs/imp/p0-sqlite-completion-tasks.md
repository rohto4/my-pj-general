## 受け入れ基準

- 表示内容と操作結果は `/api/bootstrap` が返す SQLite データに一致する。
- 画面通信失敗時に、メモリ上の仮候補や仮の状態遷移を作らない。
- Slack はアプリ内に認証情報を持たず、connector から渡された実メッセージ payload の取り込みだけを受ける。
- knowledge-vault は管理画面で有効にした対象だけを実スキャンする。

## 実装結果

- 処理フローの表示名は「タスク候補総数」「未処理の比率」「不足項目あり」「予定化の分量」「用意済待機タスク」に統一した。
- ロール、AI確定方針、knowledge-vault取り込み対象は `settings` テーブルの `admin.*` 設定として保存する。
- タグは作成と表示・非表示切替を SQLite に保存する。
- Slack はアプリ内の空 payload を送る動作を廃止し、connector から取得した実メッセージ JSON を管理画面から渡して取り込む方式にした。
- 初期デモ候補と Misskey の仮データは既存 SQLite から一度だけ削除する移行を入れた。
- 画面レビューで判明した旧 `grid-column` 競合を除去した。
- 入口別の量は候補の有無ではなく SQLite の `sources` を基準に描画し、Web手入力、Slack、knowledge-vault、Misskey を0件でも表示するようにした。
- 候補の種類は固定の種類マスタ順に描画し、TODO、検討、アイデア、予定候補、気になる事を0件でも表示するようにした。
- 横断ダッシュボードは、処理フローを上段の左2列に置き、下段を「入口別の量」「候補の種類」「優先確認 + 判断ログ」の3列に分ける配置へ変更した。左ナビを194pxへ圧縮し、各サマリを独立した列として扱う。
- 3列の横幅は固定pxではなく、利用可能幅を `1fr / 1fr / 2fr` で分配する。gapとpaddingを除いた幅で入口別の量25%、候補の種類25%、優先確認50%相当を保つ。
- 作業者用 / タスクサマリは `11fr / 9fr`、確認待ちキューは `1fr / 2fr` の可変比率へ統一した。表・操作ボタンの可読性を守れない中間幅では、確認待ちキューを1列に切り替える。
- 確認待ちキューの4フィルターは、各220pxから255pxの固定幅で左寄せにした。
