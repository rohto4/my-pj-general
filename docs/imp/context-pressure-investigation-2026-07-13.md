# コンテキスト自動圧縮の圧迫要因調査

## 目的と安全境界

「短く見える直近の会話内容でも自動圧縮が頻発する」現象を、初期読込、実入力 token、圧縮置換履歴、画像・添付、tool 出力の別々の証拠で調べた。本文、画像そのもの、Cookie、token、secret、env は読まず、ローカル session JSONL の構造・件数・サイズ・token メタデータだけを集計した。

作業開始前の保全は commit `d9b56eafb5ae8da5638f40cebe27bc433c67cd88`（`feat: consolidate P0 frontend and operations handoff`）を `rohto4` の `origin/main` へ push 済みである。

## 実施した確認

| 確認 | 方法 | 判定に使ったもの |
| --- | --- | --- |
| 初期読込の負荷 | 必須初期化ファイルの行数・byte・token実測を確認 | 751行、約44KB、約13,472 token |
| sessionログ健全性 | JSONLをストリーミング parseし、内容を表示せず件数とparse errorだけ集計 | 9,715行、423.4MiB、parse error 0 |
| 圧縮の発生点 | 各`compacted`直前の`last_token_usage`と`model_context_window`を対応付け | 19回の圧縮と入力占有率 |
| 圧縮後の残留payload | `replacement_history`内の`input_image`と`input_text`のブロック数・JSON構造サイズを集計 | 画像が圧縮後にも残留するか |
| 大きい記録の種類 | root record、nested type、画像・添付・tool関連の構造フラグを集計 | 画像・添付・toolの相対的な寄与 |

再現可能な集計器は `.agents/skills/audit-context-pressure/scripts/summarize-session.mjs` として保存する。引数にsession JSONLを明示し、raw内容を出力しない。

## 実測結果

### 初期読込は主因ではない

必須初期化の 751行・約44KB は約13,472 tokenであり、256K基準の約5.3%である。現在の実行窓258,400 tokenに対しても約5.2%に相当する。役割別ガイドのとおりリンク先を自動・再帰読込しない限り、初期化だけで圧縮を起こす量ではない。

### 圧縮直前は実入力が窓の上限近くまで達していた

sessionログには圧縮19回があり、圧縮の直前に観測できた入力 token は次のとおりだった。

| 範囲 | 入力 / 窓 | 占有率 |
| --- | --- | --- |
| 圧縮 #1〜#13 | 307,646〜329,471 / 353,400 | 87.1〜93.2% |
| 圧縮 #15〜#19 | 230,255〜243,012 / 258,400 | 89.1〜94.0% |
| 最大 | 329,471 / 353,400 | 93.2% |
| 調査時の最新入力（2026-07-13T13:36:59Z） | 198,124 / 258,400 | 76.7% |

圧縮 #14だけは直前の記録が243,582 / 353,400（68.9%）だった。その直後に実行窓が258,400へ切り替わっているため、この1点を「安全だった」根拠には使わない。累積 token 使用量は現在の入力占有率ではないため、表には使っていない。

### 高解像度画像・添付が圧縮後も置換履歴に残っていた

| 圧縮置換履歴 | #1 | #6 | #13以降（最終） |
| --- | ---:| ---:| ---:|
| `input_image` block数 | 3 | 22 | 30 |
| 画像blockの構造サイズ | 1.03MiB | 16.83MiB | 17.19MiB |
| `input_text`文字数 | 83,600 | 135,355 | 172,960 |

19回分の置換履歴に現れた画像block構造サイズの合計は263.5MiBである。同じ画像が後続の圧縮で再び数えられるため、この合計を最終的なモデル token量として扱わない。しかし最終圧縮でも30画像・17.19MiBが残っており、圧縮が画像payloadを十分に捨てず、以降の会話にも圧迫要因を持ち越していたことは確認できる。

### tool出力も副次的な圧迫要因

root recordでは圧縮記録が255.1MiB、画像関連フラグが349.1MiB、添付関連フラグが358.0MiB、tool関連フラグが87.8MiBだった。これらは同一recordを重複して数える分類であるため、加算して総量にはしない。画像・添付が最大の持続要因であり、全画面JPEG、画像生成結果、巨大なtool出力を連続させたことが圧縮頻発を増幅したと結論づける。

## 原因の結論

1. 初期読込ファイル自体は約5%で健全であり、主因ではない。
2. 同一sessionで多数のスクリーンショット・画像生成・添付を扱うと、画像payloadが圧縮後の`replacement_history`にも残る。
3. 圧縮は履歴を短くするが、今回の画像payloadを十分に除去しなかった。そのため直近の発話が短くても、実入力は窓の87〜94%へ早く戻った。
4. 巨大tool出力、貼り付けログ、全HTML/全ソースの読込は、画像に加わる副次的な圧迫要因である。
5. よって「長い会話なら必ず即座に移動」ではない。圧縮1回後に画像・長大出力を伴う局面へ入る前、または圧縮2回以上で、docs/Git断面を閉じて新sessionへ移るのが適切である。

## 採用した予防策

- 現行モデル窓に対する実入力の比率を最優先で見る。75%で切替準備、85%以上または圧縮2回以上で切替必須とする。
- 画像レビュー・画像生成・全ログ調査は、代表資料を絞り、結論をdocsへ固定してから次の実装局面へ進む。
- 設計書は対象見出し・対象ID・不足証拠だけを読む。リンク先全読込やfork全体読込で解決しない。
- 実装先行の場合も、対象機能の境界だけを設計書へ補い、全履歴を読み直さない。
- セッション切替時は`docs/imp/imp-tasks.md`、`docs/imp/imp-comp.md`、必要時のdiary、テスト済みGit断面を先に同期する。

具体的な閾値とパターン別手順は `docs/guide/context-pressure-session-guideline.md` を正本とする。

## 限界

- JSONLの構造byteとモデルtokenは別の単位である。サイズ値はpayloadの残留・相対比較にだけ使う。
- sessionログはこの1セッションの実測であり、すべてのモデル・クライアントの圧縮アルゴリズムを断定しない。
- それでも、圧縮直前の実入力tokenと置換履歴の画像残留を同時に確認したため、本sessionで圧縮が頻発した直接的な運用要因を説明するには十分な証拠である。
