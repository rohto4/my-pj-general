# 要件回答メモ 2026-06-29

## 目的

この文書は、`docs/imp/user-requirements-questionnaire-2026-06-29.md` に対するユーザー回答を、次の設計作業へ直接使える形に整理した記録である。

## 回答原文ベース整理

### A. 初期版の中心価値

- `AA-00`
  - 初期版の主利用者は `自分のみ`
  - ロールは用意するが、自分以外のロールは当面仮置きでよい

- `AA-01`
  - 最も解決したい困りごとは、`思いつきや考えごとをタスクへ昇格させて着手するまで、頭の中で管理し続ける必要があること`

- `AA-02`
  - 継続利用の最低ラインは次
  - アイデアや思いつきの登録が煩雑でないこと
  - 思いつきプールからタスク化する流れが綺麗であること
  - タスク化後も煩雑でないこと
  - ある程度ちゃんとした自動化が入っていると望ましい

- `AA-03`
  - 初期版で外したくない流れは `入力 -> 入力済みプールの理解 -> TODO 化`
  - TODO 化後の予定化は 2 段階で考える
  - 半自動:
    - ユーザーが GO サインを出す
    - タスクに必要な情報を揃える
  - 全自動:
    - GO サインなしで AI が必要情報を埋める
  - 初期版では `半自動` を優先実装でよい

### B. 正本 object と昇格ルール

- `AA-04`
  - `アイデアカード` と `タスク` は `別 object`

- `AA-05`
  - `検討事項` と `気になっている事` は `分けたい`
  - 属性はタグ管理で増えてもよい

- `AA-06`
  - 1 つの入力から `タスク` と `予定` の両方が生まれることは `許容`

- `AA-07`
  - `アイデア -> タスク` の最低条件は `特になし`
  - ユーザーがタスク化したいと思った時に GO
  - 必要情報はカレンダー等を読み取って自動生成させる

- `AA-08`
  - `タスク -> 予定` の最低条件も `特になし`
  - カレンダーや周辺情報の読み取りで十分
  - 欠ける部分は AI に提案させ、ユーザーが訂正記入する

### C. 外部入口と同期

- `AA-09`
  - やりたい順:
    - `knowledge-vault`
    - `手入力`
    - `Slack`
  - ただし実装順は `手入力先行` でよい

- `AA-10`
  - Slack は `気になった記事へのリアクション` を拾えると嬉しい
  - 興味関心の事柄をプールしたい
  - 手入力は Web で行う前提
  - Slack では気になっているキーワードや、雑に投稿した内容を蓄積するイメージ

- `AA-11`
  - knowledge-vault は範囲が広いので、初期は次に絞る案
  - `daily`
  - `imp*`
  - `task*`

- `AA-12`
  - 複数入口で同じ内容が入っても `被ってよい`
  - ユーザーが必要に応じて消す

- `AA-13`
  - 表示上は出典不要
  - ただし内部的には追いたい
  - `Slack由来` や `手入力由来` のような単純な入口種別は必須ではない
  - `URL起点の参考リンク` のような実質的な出典は保持したい

### D. AI 分別と自動化

- `AA-14`
  - 初期方針は `分別タグマスタを作り、AI に選ばせる`
  - さらに AI の案を出したうえで、`登録ボタンを押すだけ` の段階を挟みたい
  - 将来的にはこの確認段階を省略する可能性あり

- `AA-15`
  - AI に最初から任せたい範囲:
  - 気になっていることのピックアップ
  - 分類
  - 要約
  - タスク案生成
  - 予定案生成
  - ただし `全てを一気通貫で自動確定` したいわけではない
  - ノンストップ自動化は少し先の話

- `AA-16`
  - 定期回収はコストが低ければ `1 分ごと` でもよい
  - ミニ PC サーバーと回線には余裕がある前提

### E. 権限と画面

- `AA-17`
  - 外部協力者は初期版では `何もできなくてよい`
  - できあがったカレンダーのモックが見られる程度で十分
  - 権限は仮置きでよい

- `AA-18`
  - 4 窓口の優先順位は `データの上流から順番`
  - つまり最優先は `書き入れ口 / 作成口`

### F. MVP と後段

- `AA-19`
  - `Google Calendar 直接登録`
    - GO サインによる自動登録なら初期採用
  - `ガントチャート`
    - ほしい
  - `キャパ管理`
    - ほしい
  - `Codex プロジェクト開始支援`
    - ほしい
  - `外部協力者向け権限`
    - 仮置きだけでよい

## 現時点で強まった判断

- 初期版は `自分専用` を主軸にしてよい
- プロダクトの中核は `思いつきプール -> TODO 化` の流れ
- `アイデアカード` と `タスク` は分離する
- `検討事項` と `気になっている事` も分ける
- 昇格条件は厳密な必須項目より `GO で進められること` を優先する
- 初期の AI フローは `自動提案 + 人の最終登録`
- 手入力先行で実装しつつ、思想上の最優先入口は `knowledge-vault`
- 重複排除は初期必須にしない
- 外部協力者権限は初期は薄くてよい

## 次に docs へ落とすべき論点

3. `knowledge-vault > 手入力 > Slack` を踏まえた入口戦略と実装順
4. Google Calendar 直接登録の初期条件
5. 初期版の画面優先順位を `書き入れ口中心` で再整理すること
